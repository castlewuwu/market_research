> 调研日期：2026年3月24日
> 数据来源：行业实践、公开资料、产品文档

---

## 一、什么是AI工作流

**定义：** AI工作流是将AI能力嵌入业务流程，实现从"人做事"到"AI辅助做事"甚至"AI自动做事"的转变。

**核心组成：**

| 组件 | 说明 | 示例 |
|------|------|------|
| **触发器** | 什么情况下启动工作流 | 定时、事件、人工触发 |
| **输入** | AI需要什么信息 | 文本、文件、数据库、API |
| **AI处理** | AI做什么 | 生成、分析、转换、决策 |
| **输出** | 结果去哪里 | 文档、系统、通知、下一个流程 |
| **人工节点** | 哪些环节需要人 | 审核、确认、补充信息 |

## 二、代码开发AI工作流

### 2.1 工作流全景图

```
需求理解 → 技术方案设计 → 编码实现 → 代码审查 → 测试 → 部署 → 维护
    ↓            ↓            ↓          ↓        ↓      ↓      ↓
  AI辅助       AI辅助        AI主力     AI辅助   AI辅助  AI辅助  AI辅助
```

### 2.2 详细工作流搭建

#### 阶段一：需求理解

**目标：** 将模糊的业务需求转化为清晰的技术需求

**工作流搭建：**

```
输入：产品需求文档/用户故事/会议记录
  ↓
AI处理：
  - 提取核心功能点
  - 识别边界条件
  - 发现遗漏场景
  - 生成技术需求清单
  ↓
输出：结构化的技术需求文档
  ↓
人工确认：PM/开发负责人审核
```

**推荐工具：**

| 工具 | 用途 | 特点 |
|------|------|------|
| ChatGPT/Claude | 需求分析 | 通用能力强 |
| Notion AI | 文档处理 | 与文档系统深度集成 |
| 飞书智能伙伴 | 企业场景 | 了解团队上下文 |

**Prompt模板：**

```
你是一位资深技术架构师。请分析以下产品需求：

【原始需求】
{需求文档}

请输出：
1. 核心功能点（按优先级排序）
2. 技术难点和风险
3. 需要进一步澄清的问题
4. 建议的技术方案（简要）
```

  ↓
输出：技术设计文档
  ↓
人工评审：架构师/技术负责人审核
```

**推荐工具：**

| 工具 | 用途 |
|------|------|
| Claude | 复杂推理、架构设计 |
| GPT-4 | 技术方案生成 |
| Draw.io + AI | 架构图绘制 |

**Prompt模板：**

```
你是一位系统架构师。请为以下需求设计技术方案：

【需求概述】
{需求概述}

【现有技术栈】

- 数据库：{技术栈}

1. 系统架构设计（含架构图描述）
2. 核心API设计
3. 数据库表设计（关键字段）

```

#### 阶段三：编码实现（AI主力）

**目标：** AI辅助/自动生成代码

**工作流搭建：**

```
输入：技术设计 + 现有代码库
  ↓
AI处理：
  - 代码补全/生成
  - 代码重构
  - Bug修复
  - 单元测试生成
  ↓
输出：代码文件 + 测试用例
  ↓
人工审核：代码审查
```

**推荐工具组合：**

| 工具 | 定位 | 核心能力 | 价格 |
|------|------|---------|------|
| **GitHub Copilot** | 代码补全 | 行内补全、多文件理解 | $10-19/月 |
| **Claude Code** | 代码Agent | 端到端开发、复杂重构 | 按API计费 |
| **Cursor** | AI IDE | 全项目理解、多文件编辑 | $20/月 |
| **通义灵码** | 代码助手 | 中文友好、阿里生态 | 免费/付费 |
| **豆包MarsCode** | 代码助手 | 字节生态、免费版够用 | 免费/付费 |

**最佳实践工作流：**

**场景1：新功能开发**

```
Step 1: 在IDE中打开相关文件
Step 2: 用自然语言描述需求（Ctrl+K 或 Cmd+K）
Step 3: AI生成代码初稿
Step 4: 人工审查和修改
Step 5: AI生成单元测试
Step 6: 运行测试，AI修复失败用例
Step 7: 提交代码
```

**场景2：Bug修复**

```
Step 1: 复制错误信息/堆栈
Step 2: 让AI分析原因
Step 3: AI提供修复方案
Step 4: 选择方案并应用
Step 5: AI生成回归测试
```

**场景3：代码重构**

```
Step 1: 选中需要重构的代码
Step 2: 告诉AI重构目标
Step 3: AI生成重构后代码
Step 4: 对比diff，确认修改
Step 5: 运行测试确保功能不变
```

**Prompt技巧：**

```
# 好的Prompt
"在UserService中添加一个getUserById方法，需要：
1. 参数校验（userId不能为空）
2. 缓存支持（使用Redis）
3. 异常处理（用户不存在抛出UserNotFoundException）
4. 添加日志
请参考现有代码风格"

# 不好的Prompt
"写一个getUserById方法"
```

## 三、产品工作流（PRD撰写）

### 3.1 工作流全景图

```
市场调研 → 需求收集 → 需求分析 → PRD撰写 → 原型设计 → 需求评审 → 迭代
    ↓          ↓          ↓          ↓          ↓          ↓        ↓
  AI辅助    AI辅助     AI辅助     AI主力     AI辅助    AI辅助   AI辅助
```

### 3.2 详细工作流搭建

#### 阶段一：市场调研

**工作流：**

```
输入：调研目标 + 关键词
  ↓
AI处理：
  - 搜索行业报告
  - 分析竞品信息
  - 总结市场趋势
  ↓
输出：调研报告
  ↓
人工确认：PM审核关键数据
```

**推荐工具：**

| 工具 | 用途 |
|------|------|
| Perplexity | AI搜索 |
| Claude + 文件分析 | 报告生成 |
| 飞书多维表格 + AI | 数据整理 |

---

#### 阶段二：需求收集

**工作流：**

```
输入：用户反馈、客服记录、问卷数据
  ↓
AI处理：
  - 提取用户痛点
  - 分类需求类型
  - 识别高频需求
  ↓
输出：需求池（优先级排序）
```

**Prompt模板：**

```
请分析以下用户反馈，提取产品需求：

{反馈内容}

```

  └── 7. 风险评估

输出：完整PRD文档
  ↓

```

| 工具 | 用途 | 特点 |
|------|------|------|

| Claude | 复杂PRD | 推理能力强 |

## 五、AI工作流搭建方法论

### 5.1 通用搭建框架

```
Step 1：识别场景
  └── 找到高频、重复、有明确规则的任务

Step 2：拆解流程
  └── 将任务拆解为最小单元

Step 3：标注AI节点
  └── 判断每个节点AI能做什么

Step 4：设计人机协同
  └── 明确哪些环节必须人工介入

Step 5：选择工具
  └── 根据需求选择合适的AI工具

Step 6：编写Prompt
  └── 设计高质量Prompt模板

Step 7：测试迭代
  └── 小范围测试，持续优化
```

### 5.2 AI节点判断标准

| 标准 | AI适合 | 人工必须 |
|------|--------|---------|
| 规则清晰度 | 高 | 低 |
| 创意需求 | 低 | 高 |
| 风险程度 | 低 | 高 |
| 决策影响 | 小 | 大 |
| 数据依赖 | 有结构化数据 | 需要主观判断 |

### 5.3 工具选择矩阵

| 任务类型 | 推荐工具 | 理由 |
|---------|---------|------|
| 代码开发 | Cursor + Copilot | 深度IDE集成 |
| 文档撰写 | Claude + Notion AI | 长文本能力强 |

## 七、总结

### 核心结论

1. **AI工作流不是替代人，而是增强人**
   - AI承担重复性、标准化任务
   - 人在关键节点把关

2. **工具选择比模型选择更重要**
   - 深度集成的工具（Cursor、Notion AI）体验更好
   - 选择适合团队的工具

3. **Prompt是工作流的核心**
   - 好的Prompt = 好的输出
   - 建立团队的Prompt库

4. **小步快跑，持续迭代**
   - 从一个场景开始
   - 验证效果后推广
